# AGENTS.md

이 파일은 이 프로젝트에서 AI 코딩 에이전트가 따라야 할 기본 작업 규칙입니다.

---

# 핵심 원칙

- 안정성, 가독성, 유지보수성을 우선합니다.
- 불필요하게 복잡한 구조를 만들지 않습니다.
- 기존 프로젝트 구조와 코드 스타일을 따릅니다.
- 똑똑해 보이는 코드보다 이해하기 쉬운 코드를 우선합니다.
- 기술 부채를 만들지 않도록 주의합니다.

---

# 새 프로젝트 시작 시 기본 세팅

새 프로젝트에서 작업을 시작할 때는 다음을 확인합니다.

1. 필요한 의존성이 있다면 설치합니다.
2. JavaScript 또는 TypeScript 프로젝트라면 ESLint 설정을 확인합니다.
3. Prettier 설정을 확인해 코드 포맷을 통일합니다.
4. 프로젝트가 정상 실행되는지 확인합니다.
5. 새 기능 구현 전에 명백한 린트/포맷 문제를 정리합니다.
6. 필요한 설정 파일이 빠져 있다면 생성하거나 제안합니다.

---

# 코딩 기준

- 함수는 하나의 역할에 집중하도록 작성합니다.
- 이름은 명확하고 설명적으로 작성합니다.
- 중복 코드를 피합니다.
- 복사 붙여넣기보다 재사용 가능한 함수나 모듈을 우선합니다.
- 사람이 읽기 쉬운 코드를 작성합니다.
- 주석은 의미 있는 설명이 필요할 때만 추가합니다.
- 과하게 설계하지 않습니다.

---

# 유지보수 규칙

프로젝트가 커져도 관리하기 쉬운 상태를 유지해야 합니다.

- 파일이 비정상적으로 커지면 사용자에게 알려줍니다.
- 파일을 읽고 수정하기 어려워지기 전에 리팩토링을 제안합니다.
- 작은 모듈로 분리해야 할 파일을 보고합니다.
- 공통 함수로 빼야 할 중복 로직을 보고합니다.
- 나중에 유지보수 문제를 만들 수 있는 구조적 문제를 보고합니다.
- 작업 중 발견한 기술 부채를 보고합니다.
- 프로젝트 구조를 개선할 기회가 있으면 알려줍니다.

중요:

- 리팩토링이 필요하면 제안합니다.
- 사용자가 명시적으로 요청하지 않은 대규모 리팩토링은 하지 않습니다.
- 기능 구현 중 관련 없는 코드는 수정하지 않습니다.

# 파일 크기 기준

아래 기준은 절대 규칙이 아니라 참고용 기준입니다.

## 검토 권장

- 단일 파일 500줄 이상

## 리팩토링 권장

- 단일 파일 800줄 이상

## 강한 리팩토링 경고

- 단일 파일 1200줄 이상

위 기준을 넘으면 다음을 보고합니다.

- 어떤 파일이 기준을 넘었는지
- 왜 유지보수가 어려워질 수 있는지
- 어떤 방식으로 분리하면 좋을지
- AI 작업 시 불필요하게 많은 토큰을 사용할 가능성이 있는지

---

# 토큰 절약 경고

AI 작업 효율과 토큰 사용량을 고려합니다.

- 너무 큰 파일은 작은 수정에도 많은 컨텍스트를 요구할 수 있습니다.
- 1000줄이 넘는 파일은 기능 단위 분리를 검토합니다.
- 3000줄이 넘는 파일은 우선적으로 분리 계획을 제안합니다.
- 5000줄이 넘는 파일은 작은 기능 수정 전에도 구조 개선 필요성을 강하게 알립니다.
- 중복 코드가 많으면 같은 내용을 반복해서 읽게 되어 토큰 낭비가 커질 수 있습니다.
- 작업 완료 보고에 토큰 낭비 가능성이 큰 파일을 포함합니다.

단, 사용자가 요청하지 않은 대규모 구조 변경은 하지 않습니다.

---

# 기능 개발 절차

구현 전:

- 기존 구조를 이해합니다.
- 재사용 가능한 코드가 있는지 확인합니다.
- 부작용이 생길 수 있는 부분을 확인합니다.

구현 후:

- 기능이 정상 동작하는지 확인합니다.
- 문법 오류를 확인합니다.
- 린트 오류를 확인합니다.
- 포맷팅 문제를 확인합니다.
- 유지보수상 우려되는 부분을 보고합니다.
- 리팩토링 기준에 가까워진 파일이 있는지 보고합니다.

---

# UI / 프론트엔드 규칙

- UI 컴포넌트는 가능한 한 작고 모듈화합니다.
- 하나의 거대한 파일에 UI, 상태, 이벤트, 스타일 로직을 모두 쌓지 않습니다.
- 재사용 가능한 컴포넌트나 유틸 함수를 우선합니다.
- 반응형 레이아웃을 고려합니다.
- 스타일 코드는 섹션별로 정리합니다.

---

# Git 규칙

- 커밋은 가능한 한 하나의 목적에 집중합니다.
- 관련 없는 변경을 한 커밋에 섞지 않습니다.
- 사용자가 요청하지 않으면 git history를 다시 쓰지 않습니다.

---

# 최종 보고 형식

중요한 작업이 끝나면 아래 형식으로 한국어로 보고합니다.

## 완료된 작업

- 구현하거나 수정한 내용

## 검증 결과

- 문법 검사 결과
- ESLint 검사 결과
- Prettier 검사 결과
- 실행 또는 페이지 확인 결과

## 주의사항

- 버그 가능성
- 성능 문제
- 유지보수 위험
- 현재는 동작하지만 나중에 문제가 될 수 있는 부분

## 리팩토링 권장사항

- 분리하면 좋은 파일
- 중복 코드
- 너무 커진 함수나 모듈
- 구조 개선 포인트
- 토큰 낭비 가능성이 큰 파일

## 다음 작업 추천

- 이어서 진행하면 좋은 작업
- 안전한 개선 순서
- 나중에 처리하면 좋은 기술 부채
